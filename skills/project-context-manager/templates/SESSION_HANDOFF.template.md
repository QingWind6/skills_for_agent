# 会话交接指南

> **给 Agent 的重要说明**：这是跨会话工作的标准流程，请严格遵守。

## 🎯 目的

本指南确保多个会话之间的开发工作能够无缝衔接，避免重复工作和信息丢失。

---

## 📥 新会话开始时（必读）

### Step 1: 快速了解项目 (5 分钟)

按顺序阅读以下文件：

1. **README.md** (2 分钟)
   - 了解项目是什么
   - 技术栈概览
   - 快速开始指南

2. **PROJECT_STATUS.md** (3 分钟) ⭐ **最重要**
   - 当前处于哪个阶段
   - 已完成什么
   - 正在做什么
   - 下一步做什么

### Step 2: 深入了解上下文 (10 分钟)

根据需要阅读：

3. **CONTEXT.md** (5 分钟)
   - 详细的开发上下文
   - 代码片段和设计思路
   - 已知问题和解决方案

4. **技术方案书.md** (可选，需要时)
   - 架构设计细节
   - 技术选型理由

5. **开发指南.md** (可选，需要时)
   - 具体实施步骤
   - 代码示例

### Step 3: 确认工作内容 (2 分钟)

检查 `PROJECT_STATUS.md` 中的：

- ✅ **"进行中任务"** - 是否有未完成的工作需要继续？
- ✅ **"待办任务"** - 下一步应该做什么？
- ✅ **"已知问题"** - 是否有阻塞问题需要先解决？

### Step 4: 开始工作

- 如果有"进行中任务"，继续完成它
- 如果没有，从"待办任务"中选择优先级最高的开始
- 开始前，在 `PROJECT_STATUS.md` 中更新"进行中任务"

---

## 📤 会话结束时（必做）

### Step 1: 更新项目状态 (5 分钟)

编辑 `PROJECT_STATUS.md`：

1. **更新"已完成任务"**
   - 将完成的任务打勾 ✅
   - 如果完成了整个子任务，标记为 `[x]`

2. **更新"进行中任务"**
   - 如果任务完成了，清空这个部分
   - 如果任务未完成，详细描述当前进度和下一步

3. **更新"已知问题"**
   - 添加新发现的问题
   - 更新已有问题的状态

4. **更新"文件结构现状"**
   - 标记新创建的文件/目录为 ✅
   - 保持准确性

5. **更新元数据**
   - 修改"最后更新"时间
   - 填写"最后更新者"（你的 Agent 名称或会话 ID）

### Step 2: 记录详细上下文 (5-10 分钟)

编辑 `CONTEXT.md`（如果有重要信息需要记录）：

1. **正在开发的功能**
   - 更新当前功能的开发状态
   - 记录设计思路和实现细节
   - 列出待解决问题

2. **重要代码片段**
   - 添加关键代码片段（如果有）
   - 说明用途和注意事项

3. **调试信息**
   - 记录遇到的问题和解决方案
   - 记录尝试过的方法

4. **技术研究笔记**
   - 记录学到的新知识
   - 添加参考资料链接

### Step 3: 代码提交（如果使用 Git）

```bash
# 提交代码
git add .
git commit -m "[模块] 简短描述

详细说明：
- 完成了什么
- 修改了哪些文件
- 还有什么未完成"

# 推送到远程（可选）
git push
```

### Step 4: 给下一个会话留言

在 `PROJECT_STATUS.md` 的"进行中任务"部分，写清楚：

- ✅ **当前进度**: 完成到哪一步了
- ✅ **下一步**: 应该做什么
- ✅ **注意事项**: 有什么需要特别注意的
- ✅ **相关文件**: 涉及哪些文件

**示例**:
```markdown
## 🚧 进行中任务

**当前任务**: 实现 FFmpeg 视频解码器

**负责人**: Agent-Session-001

**开始时间**: 2026-03-10 14:00

**预计完成**: 2026-03-10 18:00

**进度说明**:
- ✅ 已完成: VideoDecoder 类的基本结构
- ✅ 已完成: open() 方法实现
- 🚧 进行中: readFrame() 方法实现（70%）
- ⏳ 待完成: seek() 方法实现

**下一步**:
1. 完成 readFrame() 方法的错误处理
2. 实现 seek() 方法
3. 编写单元测试
4. 测试不同视频格式

**遇到的问题**:
- FFmpeg 的 av_read_frame() 偶尔返回 AVERROR_EOF，需要处理
- 参考资料: http://dranger.com/ffmpeg/tutorial01.html

**相关文件**:
- src/video/decoder.h
- src/video/decoder.cpp
```

## 🎨 最佳实践

### 1. 保持文档同步

- 代码变更后立即更新文档
- 不要等到会话结束才更新
- 小步提交，频繁更新

### 2. 详细记录问题

- 遇到问题立即记录到 `CONTEXT.md`
- 包含错误信息、尝试的解决方案
- 记录最终的解决方法

### 3. 清晰的任务描述

- 任务描述要具体，不要模糊
- 包含验收标准
- 列出相关文件路径

### 4. 代码注释

- 关键代码添加注释
- 复杂逻辑说明设计思路
- FFmpeg、OpenGL 等 API 调用说明用途

### 5. 测试驱动

- 每完成一个模块立即测试
- 记录测试结果到 `CONTEXT.md`
- 发现问题立即修复或记录

## 🚨 常见错误

### ❌ 错误 1: 不读文档就开始工作

**后果**: 重复已完成的工作，或者方向错误

**正确做法**: 先花 10-15 分钟阅读文档，了解当前状态

### ❌ 错误 2: 会话结束不更新文档

**后果**: 下一个会话不知道做了什么，信息丢失

**正确做法**: 会话结束前必须更新 `PROJECT_STATUS.md`

### ❌ 错误 3: 任务描述不清晰

**后果**: 下一个会话不知道从哪里继续

**正确做法**: 详细描述当前进度、下一步、相关文件

### ❌ 错误 4: 不记录遇到的问题

**后果**: 下一个会话可能遇到同样的问题，浪费时间

**正确做法**: 立即记录问题和解决方案到 `CONTEXT.md`

### ❌ 错误 5: 一次做太多事情

**后果**: 会话结束时任务未完成，难以交接

**正确做法**: 小步迭代，每次完成一个小模块

## 📞 紧急情况处理

### 情况 1: 文档不一致

如果发现 `PROJECT_STATUS.md` 和实际代码不一致：

1. 检查代码的实际状态
2. 更新 `PROJECT_STATUS.md` 以反映真实情况
3. 在 `CONTEXT.md` 中记录发现的不一致

### 情况 2: 无法理解上一个会话的工作

如果无法理解上一个会话做了什么：

1. 检查 Git 提交历史（如果有）
2. 阅读代码注释
3. 在 `CONTEXT.md` 中记录困惑
4. 必要时重构或重写不清晰的代码

### 情况 3: 发现严重 Bug

如果发现严重 Bug：

1. 立即记录到 `PROJECT_STATUS.md` 的"已知问题"
2. 标记为"严重程度: 高"
3. 在 `CONTEXT.md` 中详细描述
4. 优先修复，或者标记为阻塞问题

## 🎓 示例场景

### 场景 1: 继续未完成的任务

**会话开始**:
1. 阅读 `PROJECT_STATUS.md`，发现"进行中任务"是"实现 FFmpeg 解码器"
2. 阅读任务描述，了解已完成 70%，还需要完成错误处理和 seek 方法
3. 打开 `src/video/decoder.cpp`，找到未完成的代码
4. 继续实现

**会话结束**:
1. 完成了 readFrame() 的错误处理
2. 完成了 seek() 方法
3. 更新 `PROJECT_STATUS.md`，标记任务为"已完成"
4. 在 `CONTEXT.md` 中记录实现细节

### 场景 2: 开始新任务

**会话开始**:
1. 阅读 `PROJECT_STATUS.md`，发现没有"进行中任务"
2. 查看"待办任务"，选择优先级最高的"实现基础窗口"
3. 更新 `PROJECT_STATUS.md`，将任务移到"进行中任务"
4. 开始实现

**会话结束**:
1. 完成了基础窗口和菜单栏
2. 更新 `PROJECT_STATUS.md`，标记任务为"已完成"
3. 添加下一个任务到"待办任务"
4. 在 `CONTEXT.md` 中记录 ImGui 的使用心得
